# Deploy plan — security audit Stage 6 mitigations

> Generated by the `security-audit-pentest` skill, 2026-05-12. The audit fixes were
> pushed in commit `4ef48e2` and made transitional in this commit. Read all steps
> before running anything.

## What's changing in production

| Surface | Change | Risk if mis-ordered |
|---|---|---|
| Firestore rules | `users/{uid}` allow-list update; new `private/*` subcollection; `events` lat/lng bucketing enforced; `processedWebhooks` denied to clients | Strict rules can reject writes from old clients still running |
| Cloud Functions | Add `requestAccountDeletion`; switch `fcmToken` reads to private subcollection (with legacy-path fallback during transition); rankings refresh cache | Functions reading from new path with no data there → broken push |
| Hosting (`firebase.json`) | Codify security headers; add CSP in Report-Only | Header regressions caught by `.github/scripts/check-prod-headers.sh` |
| Web bundle (`app/dist/`) | Account deletion now calls Cloud Function; `fcmToken` written to subcollection; email no longer persisted to `users/{uid}` | Old tabs still running old bundle for ~24h |
| `app/app.json` | App Links / Universal Links + new domain `cambiafiguritas.online` | Requires hosting `.well-known/` files |

## Transitional state — what's safe to deploy NOW

This branch is **transitional**. Both code paths and rule paths accept the legacy
schema in addition to the new one:

- `functions/src/notifications.ts:readFcmToken` reads from BOTH paths (prefers new).
- `functions/src/digest.ts` scans both `collectionGroup('private')` and legacy `users.fcmToken`, dedupes by uid.
- `functions/src/payments/entitlement.ts` reads thank-you push token from BOTH paths.
- `firestore.rules` `users/{uid}` create rule does NOT reject `email`/`fcmToken` (old clients still write them).
- `firestore.rules` `users/{uid}` update allow-list **includes** `email`/`fcmToken` (old client updates still work).

This means the deploy is **forward-compatible with both old and new clients** simultaneously.

## Deploy order

Run from the repo root. **Verify each step before continuing.**

### 0. Pre-flight (local)

```bash
# Confirm working tree matches what we plan to deploy
git log --oneline -1
git status

# TypeScript clean
( cd functions && npm install --no-audit --no-fund && npm run build )
( cd app && npm install --no-audit --no-fund && npm run typecheck )

# Rule tests (requires firebase-tools + java)
( cd firestore-tests && npm install && npm test )

# Web bundle freshly built
( cd app && npm run build:web )
ls -la app/dist/  # should include index.html, _expo/, manifest.json, etc.
```

### 1. Deploy Cloud Functions first

Functions accept BOTH legacy and new fcmToken paths, so deploying them first does
not break any user. The new `requestAccountDeletion` callable becomes live but is
only invoked by the new web bundle, which isn't deployed yet.

```bash
firebase deploy --only functions --project cambiafiguritas
```

Smoke test:

```bash
# requestAccountDeletion is now listed; tilopay / play paths unchanged
firebase functions:list --project cambiafiguritas | grep -E 'requestAccountDeletion|confirmTilopayPayment'
```

### 2. Deploy hosting

This rolls out the new SPA bundle. Web users get the new behaviour as the CDN
propagates (≈ 1–5 min). Open tabs keep running the old bundle until refresh.

```bash
firebase deploy --only hosting --project cambiafiguritas
```

Smoke tests:

```bash
# Headers — should now include codified set; CSP-Report-Only present.
bash .github/scripts/check-prod-headers.sh

# OG endpoints respond
curl -sI https://cambiafiguritas.online/ | head -5
curl -sI https://cambiafiguritas.online/u/$(uuidgen) | head -5
```

### 3. Deploy Firestore rules

Rules are still permissive (transitional). New clients write to the private
subcollection; old clients write to the legacy field — both accepted.

```bash
firebase deploy --only firestore:rules --project cambiafiguritas
```

Smoke tests (use the existing emulator harness against staging or test data,
NOT production):

```bash
cd firestore-tests && npm test
```

### 4. Migrate existing fcmTokens

After functions/hosting/rules are live and stable for ≥ 1 hour:

```bash
cd functions
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json
npx ts-node scripts/migrate-fcm-tokens.ts --dry-run   # preview the move
# Inspect output. When happy:
npx ts-node scripts/migrate-fcm-tokens.ts             # actually move them
```

The script writes a `migration-backup-<timestamp>.json` next to the working dir
so the move is reversible.

### 5. Host App Links / Universal Links verification files

Without these, `autoVerify=true` in `app.json` won't pass and Android still
prompts. The web layer needs them at known paths.

Add into `app/web/.well-known/` (then re-run `npm run build:web` and redeploy
hosting):

- `assetlinks.json` — Android App Links. Pattern:

  ```json
  [
    {
      "relation": ["delegate_permission/common.handle_all_urls"],
      "target": {
        "namespace": "android_app",
        "package_name": "com.cambiafiguritas.app",
        "sha256_cert_fingerprints": [
          "<obtained via `eas credentials` or Play Console → Setup → App integrity>"
        ]
      }
    }
  ]
  ```

- `apple-app-site-association` — iOS Universal Links. Pattern:

  ```json
  {
    "applinks": {
      "apps": [],
      "details": [
        {
          "appID": "<TEAM_ID>.com.cambiafiguritas.app",
          "paths": ["*"]
        }
      ]
    }
  }
  ```

After deploy, verify:

```bash
curl -sI https://cambiafiguritas.online/.well-known/assetlinks.json | head -3
curl -sI https://cambiafiguritas.online/.well-known/apple-app-site-association | head -3
```

### 6. Out-of-band data audit

```bash
# 1. Identify users who used the dev devCompleteOrder path on production. Should be 0 if
```

### 7. Tighten transitional rules (FOLLOW-UP COMMIT, NOT NOW)

After ≥ 7 days of zero traffic from legacy clients writing `email` / `fcmToken` to
the main doc, open a follow-up PR that:

- Re-adds `!('email' in request.resource.data)` and `!('fcmToken' in request.resource.data)`
  to the `users/{uid}` create rule.
- Removes `email` and `fcmToken` from the `users/{uid}` update allow-list.
- Removes the legacy-path fallback in `functions/src/notifications.ts:readFcmToken`,
  `digest.ts`, and `payments/entitlement.ts`.
- Re-enables the commented-out tests in `firestore-tests/rules.test.mjs`.

Track legacy traffic in Cloud Logging:

```
resource.type="cloud_firestore_database"
protoPayload.methodName="google.firestore.v1.Firestore.Commit"
protoPayload.request.writes.update.fields.fcmToken:*
```

Zero hits for 7 days → safe to tighten.

### 8. Switch CSP from Report-Only to enforcing

After ≥ 7 days of CSP report monitoring with zero legitimate violations:

```diff
- { "key": "Content-Security-Policy-Report-Only", "value": "default-src 'self'; ..." }
+ { "key": "Content-Security-Policy",             "value": "default-src 'self'; ..." }
```

Sentry can ingest CSP reports; configure `report-uri` if you want centralised review.

### 9. Replace AdMob test publisher IDs

In `app/app.json`:

```diff
- "androidAppId": "ca-app-pub-3940256099942544~3347511713",
- "iosAppId":     "ca-app-pub-3940256099942544~1458002511",
+ "androidAppId": "<your real AdMob app id>",
+ "iosAppId":     "<your real AdMob app id>",
```

The `~3940256099942544` is Google's well-known test/sample id. Shipping it to
production runs test ads and pays nothing.

## Rollback

| Layer | Rollback |
|---|---|
| Functions | `firebase deploy --only functions --project cambiafiguritas` after `git checkout 94e5fe0 -- functions/` |
| Hosting | Firebase Hosting → release history → restore previous release in console, or `firebase hosting:clone source live:<previous> target live` |
| Rules | Use the Firestore rules history in console to revert to the previous rule set (atomic). |
| fcmToken migration | Re-import from the `migration-backup-<timestamp>.json` produced by the script. |

## Out-of-scope for this deploy (do later)

- Full split of `users/{uid}` into public + private docs (currently only `email`/`fcmToken`
  moved). Read rule still allows any signed-in user to read the rest of the user doc
  (name, city, lat/lng, premium, etc.). For full F-DB-001 closure, redesign the
  client-side matching algorithm to use a separate public-profile collection.
- `cambiafiguritas-staging` Firebase project for safer pre-prod testing of rule/function
  changes. Today the team deploys directly to `cambiafiguritas`.
